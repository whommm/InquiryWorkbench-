企业级智能采购工作台 | 建设规划方案

适用对象：产品研发部、采购管理部、公司管理层
1. 建设背景与核心价值
当前企业采购业务正面临从“粗放式询价”向“精细化数据运营”转型的关键期。现有的线下 Excel 作业模式存在以下核心痛点：
需求清单复杂：客户提供的询价单格式非标，常包含“指定品牌批量询价”与“无品牌/任意品牌”混合的情况，人工拆解耗时。
报价合规风险：供应商报价规格与需求规格常有细微出入（如替代型号），人工录入易忽略备注，导致后续交付风险。
数据资产流失：大量优质供应商的联系方式分散在个人微信/聊天记录中，缺乏关键字段（手机号）的结构化沉淀，导致数据无法被团队复用。
本项目旨在构建一套全流程智能采购协作平台。通过 AI 自动化处理复杂单据，通过刚性代码逻辑确保比价合规，并以“有效联系方式”为锚点沉淀企业级供应链资产。

2. 核心业务流程与功能定义
系统设计遵循 “AI 辅助录入 + 规则引擎风控 + 数据有效沉淀” 的闭环逻辑。

2.1 复杂清单的智能感知与映射 (Smart Mapping)
针对客户原始询价单中“多品牌混排”或“无品牌”的情况，系统具备深度的上下文理解能力：
批量品牌识别：当 AI 接收到供应商针对“西门子”批次的报价时，能自动检索表格中所有品牌要求为“Siemens”的行，进行批量匹配，而非逐行操作。
开放品牌处理：针对需求列中“无品牌要求”的物料，系统强制提取供应商报价中的实际品牌并填入“供方品牌”列，确保比价基准明确。
现状感知：AI 在写入前会先读取表格当前的填充率。若发现某行已有低价，会提示“当前报价高于现有最低价”，辅助决策。
2.2 规格差异风控机制 (Spec Compliance Check)
为杜绝“错型发货”风险，系统内置严格的规格校验逻辑：
差异检测：AI 自动比对 需求规格 与 供方报价规格。
场景：需求“GR28”，报价“GR38”。
自动备注：一旦发现文本不完全一致，系统将：
在“备注”栏自动写入差异说明（如：“供方型号为 GR38”）。
在前端高亮该单元格，提示采购员进行人工确认。
此过程不阻断录入，但强制留痕。
2.3 动态竞价排位引擎 (Competitive Ranking)
贯彻“低价优先”的采购原则，系统摒弃固定的列位填充，采用后端刚性排序算法：
排序逻辑：Slot 1 (左) < Slot 2 (中) < Slot 3 (右)。
执行机制：无论录入顺序先后，系统实时计算当前行所有有效报价，始终将最低价置于最左侧推荐位，直观展示最优解。

3. 供应链数据资产体系 (Supply Chain Assets)
数据沉淀是本项目的核心战略价值。我们重新定义了“有效供应商数据”的标准，确保数据的可复用性。
3.1 供应商主数据标准 (Supplier MDM)
只有具备“可联系性”的数据才具有商业价值。系统强制执行以下数据录入标准：
核心三要素（Mandatory）：
供应商全称：对接企查查/天眼查数据，清洗为标准公司名。
联系方式（关键）：必须包含手机号。这是后续数据复用、一键拨号、微信添加的基础。无手机号的记录将被标记为“低质量数据”。
数据归属人：记录该供应商是由哪位采购员引入的（User ID）。
辅助要素（Optional）：联系人姓名、职位等。
3.2 历史价格向量库 (RAG Knowledge Base)
模糊匹配复用：针对工业品名称不规范的特点（如“电机” vs “马达”），利用向量数据库进行语义检索。
寻源推荐：当新询价单导入时，系统自动匹配历史库。如果发现某物料曾由“张三”在“2024-10”从“139xxxx8888”处采购过，直接在界面右侧推荐该供应商及联系方式，实现团队资源共享。

4. 技术架构与实施方案
4.1 系统架构设计
前端 (Interaction)：采用 React + 电子表格引擎，支持复杂的冻结列、多行批注及差异高亮显示，还原 Excel 高效操作体验。
中台 (Logic)：
Python 规则引擎：负责价格排序、规格比对、数据清洗等核心业务逻辑，确保准确性。
LLM 服务层：负责非结构化报价文本的解析与意图识别。
数据 (Storage)：
PostgreSQL：存储业务数据及供应商主数据。
pgvector：存储物料描述的特征向量，支撑高维模糊检索。
4.2 实施阶段规划
第一阶段（比价提效）：上线 Excel 智能解析、AI 辅助录入、规格差异自动备注功能。
第二阶段（资产沉淀）：建立供应商主数据强制录入规范（手机号必填），完成历史数据清洗入库。
第三阶段（全域协同）：打通 RAG 向量检索，实现“上传即推荐”，激活企业沉睡的供应链数据资产。

5. 预期收益
效率提升：复杂询价单处理时间预计缩短 60% 以上，AI 自动完成繁琐的录入与比对工作。
风险降低：通过规格差异自动备注与低价强制排序，显著降低错采、贵采风险。
资产增值：将分散在员工个人手中的供应商资源转化为企业级资产，实现“一人寻源，全员复用”。


配置需求：
服务器：8C / 8-16G / 500G SSD (视实际情况决定)。
RAG：BGE-M3 （本地或者api都可行，本地速度快，占用约2G显存，纯cpu推理也可）。
LLM：推荐阿里千问或者智谱GLM API。本地部署想要达到理想效果至少需要一张A100，价格几万至十几万不划算。
首年投入预估：
服务器硬件：首先确认公司是否已有可用服务器，视数据量决定机器内存大小，可参考现有BDP服务器配置，云服务器选择阿里云等云服务厂家。选择物理机可以自己寻找i5i7cpu和内存条以及硬盘组装性价比高。
API 费用：~2,000 元/年（大致估算，具体要按照平均每日token用量评估，有优化空间。）

费用投入预估

方案A：复用公司已有服务器（需要确认机器配置），使用千问或者智谱api，预计年api费用不超过2000元。
方案B：另行购买云服务器，8C一般够用，内存需求由数据量决定，可尝试8G或16G或者更高，年约3-5K左右，算上api费用5-7K。
方案C：使用本地物理机作为服务器（物理机一次性费用，但需要支付电费并维护数据安全）（以下设备仅来自网络推荐，可以由公司专业人员选型）：

附言：
目前只是一份大致的框架，后续有很大优化空间，
比如1：可以使用多模态模型转译图片与语音内容而非纯文本交互；
2：可以使用企业微信官方bot接口，为每个员工账号配置一个bot账号，实现ai通过企业微信或者微信群聊的方式进行自动询价。
（企业微信api功能强大，支持图片发送、文件发送、获取文件、设定日程提醒等等功能）。
3：可以配置多源搜索工具，对于无历史商家或者已有商家不报价情况进行多源搜索厂家。
4：后期可尝试配置浏览器自动化工具，让ai模拟人类真实操作浏览器作业（可能不太稳定，距离落地稳定运行需要大量优化测试）
5：可尝试使用阿里云等提供的语音或者短信api接口，自动拨号发短信询价（逻辑可行，未尝试确认，不确定接口是否可用）
6：可以配置邮件管理工具，ai自动发送询价邮件。
其他可能集成功能后续再视情况补充。
